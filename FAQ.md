## 1. 基本編：ライセンス構成を理解する

### Q1-1. HESTIA はオープンソースソフトウェアですか？

はい、**License A として GNU Affero General Public License version 3.0（AGPL-3.0）を採用**しており、AGPL-3.0 は OSI（Open Source Initiative）承認のオープンソースライセンスです。

ただし、本ソフトウェアはトリプルライセンス構造を取っており、**非商用利用者にのみ AGPL-3.0 が提供され、商用利用者は License B または License C に従う**必要があります。

### Q1-2. なぜ3つのライセンスがあるのですか？

非商用コミュニティにはオープンソースとしての自由を提供しつつ、商用利用によるプロジェクトへの貢献（コードの公開または金銭的貢献のいずれか）を確保し、プロジェクトの持続可能性を担保するためです。

- **License A（AGPL-3.0、非商用者向け）**：完全に無償、オープンソース。個人・研究・NPO等のコミュニティを支援
- **License B（相互主義、商用無償者向け）**：商用利用は無料だが、成果物をオープンソースとして公開することで貢献
- **License C（商用サブスク、商用有償者向け）**：金銭的対価を支払うことで貢献し、成果物は非公開可、優先サポート付

### Q1-3. MySQL や Redis と同じ形ですか？

非常に近いです。MySQL は GPL+商用、Redis は（2025年時点で）AGPL+RSAL+SSPL のトリプルライセンスを採用しています。本プロジェクトは、Redis や Elastic が2024〜2025年に採用した AGPL を含むマルチライセンス構造と同じ考え方に立っています。

### Q1-4. 商用禁止のオープンソースですか？

**いいえ、商用利用は禁止されていません**。商用利用も完全に許可されていますが、その場合は License A（AGPL-3.0）ではなく License B または License C を選択する必要があります。

この点は、AGPL-3.0 そのものが商用利用を禁じていないこと（自由4条すべてを保証）と整合しています。本プロジェクトが商用利用者に AGPL-3.0 を提供しない決定をしているだけです。

## 2. 利用判定編：自分はどのライセンスか

### Q2-1. 個人で趣味のプログラミングに使います。どれですか？

**License A（AGPL-3.0）** です。個人の趣味・学習・自己研鑽は LICENSE.md 第0章0.1に定める「商用利用」に該当しません。

### Q2-2. 大学の研究室で学術研究に使います。どれですか？

**License A（AGPL-3.0）** です。LICENSE.md 第0章0.2(b) により、教育機関・研究機関による非営利目的の使用は商用利用から除外されます。

### Q2-3. NPOで使います。資金源は補助金です。どれですか？

**License A（AGPL-3.0）** です。LICENSE.md 第0章0.2(b) により、慈善団体による非営利目的の使用は、資金源を問わず商用利用から除外されます。

### Q2-4. スタートアップで開発中の製品に組み込む予定ですが、まだ売上はありません。どれですか？

**License B または License C** です。「売上の発生が見込まれる」場合も商用利用に該当するため（LICENSE.md 0.1(b)）、売上ゼロでも事業計画書に記載されていれば商用です。

ただし、年間総売上高が **所在地に応じて 1,000万円（日本企業）または 50,000米ドル（日本以外の企業）** 未満の場合は、LICENSE.md 第0章0.2(d) の小規模事業者救済により、自己申告手続きを経て License A として利用可能です。

### Q2-5. 営利企業の社員ですが、まずは自分のPCで評価してみたいです。どれですか？

**License A（AGPL-3.0）**（評価期間中）です。LICENSE.md 第0章0.2(c) により、本番環境で稼働させない評価・検証目的の使用で、開始から180日以内であれば商用利用から除外されます。

180日経過後も評価を継続する場合、または本番環境での使用を開始する場合は、License B または License C に切り替える必要があります。なお、この評価期間救済は前回の使用終了から12ヶ月経過後にのみ再利用できます（無限ループ防止）。

### Q2-6. 営利企業の社内業務効率化のために使います。外部への売上は発生しません。どれですか？

**License B または License C** です。売上が直接発生しなくても、本ソフトウェアの使用により業務効率化（＝経済的利益）が生じるため、商用利用に該当します。LICENSE.md 第0章0.1にいう「売上」には「金銭的価値のある対価」が含まれるため、業務効率化による利益は商用の範疇です。

ただし、Q2-4と同様、小規模事業者救済（0.2(d)）または評価期間救済（0.2(c)）が適用される場合があります。

### Q2-7. 受託開発で顧客の製品に組み込みます。私は個人事業主です。どれですか？

**License B または License C** です。受託開発業務は LICENSE.md 第0章0.1(b)(iii) により明確に商用利用です。誰に売上が立つか（あなた／顧客）を問いません（0.3条）。

顧客が License C のサブスクライセンシーである場合、あなたも「受託業務提供者」として同サブスクの範囲でカバーされることが一般的です。顧客と事前確認してください。

### Q2-8. 「商用」と「非商用」がグレーです。どうすれば？

LICENSE.md 第0章0.4 に基づき、**使用開始前に書面でライセンサーに照会**してください。回答を得るまでは License A または License C に基づく利用に限定する必要があります。

## 3. AGPL-3.0 義務編：非商用利用者の義務

### Q3-1. AGPL を使うと、私のアプリ全体のソースを公開しないといけないのですか？

**いいえ。これはAGPLについて最もよくある誤解です。**

AGPL-3.0 が公開を要求するのは、**本ソフトウェア自体（およびその改変部分）のソースコード**であり、あなたのアプリケーション全体ではありません。

例：本ソフトウェアを呼び出す形でクラウドアプリを構築した場合、そのアプリのビジネスロジック・ドメインモデル・UI 等は公開不要です。本ソフトウェア自体を改変してネットワーク経由で提供した場合のみ、その**改変部分**の公開義務が生じます。

### Q3-2. 本ソフトウェアを「使う」だけで公開義務が発生しますか？

**いいえ。未改変の本ソフトウェアを使用するだけでは、ソースコードの公開義務は発生しません**。

AGPLのソース公開トリガーは以下の2つです。

1. **配布**：改変したバイナリや改変したソースコードを第三者に渡す場合
2. **ネットワーク経由の提供**（AGPL 第13条）：改変版を自社サーバーで動かし、その機能をネットワーク経由で第三者に提供する場合

「使用するだけ」では両方のトリガーに該当しません。

### Q3-3. 社内で改変して、社内の同僚だけが使います。公開義務はありますか？

**原則として発生しません**。AGPL の「ネットワーク経由の提供」は、ライセンス文言上「ユーザーに提供する」ことを指し、**社内の同一法人内での提供はこれに含まれない**と一般に解釈されています（ただし、法人を越えた共有、関連会社への提供は「第三者への提供」に該当する可能性あり）。

ただし、社内利用であってもそれが**商用利用に該当する場合**（Q2-6参照）、そもそも License A を選択できず License B または License C になります。ご注意ください。

### Q3-4. 本ソフトウェアを改変してSaaSを作り、無料で公開します。公開義務は？

**はい、改変部分の公開義務が発生します**。

AGPL第13条により、改変版をネットワーク経由でユーザーに提供する場合、そのユーザーに対して改変部分のソースコードへのアクセスを目立つ形で提供しなければなりません。具体的には以下のいずれかが必要です。

- アプリケーション内に「Source Code」リンクを表示し、GitHub等で改変版ソースを公開
- API応答のヘッダーにソースコードURLを含める
- ユーザーマニュアルまたはフッターに明示的にソース入手方法を記載

### Q3-5. 改変版をSaaSとして提供する場合、有料であっても License A（AGPL）で進められますか？

**いいえ、できません**。SaaSとして対価を得る時点で商用利用（第0章0.1）になるため、License A は選択できず、License B または License C を選ぶ必要があります。

### Q3-6. バグ修正のPull Requestを本プロジェクトにも送るつもりです。それでも自分のコードを公開する必要はありますか？

本プロジェクトにPRを送る場合、そのPRコード部分は CLA を通じてライセンサーに再ライセンス権を付与することになります。本ソフトウェア自体の改変をコミュニティに還元する行為なので、AGPLの趣旨に沿った理想的な利用方法です。

一方、あなたのアプリケーション側のコード（本ソフトウェアを呼び出す側）は、Q3-1のとおり公開不要です。

## 4. 成果物編：License B の公開義務を理解する

### Q4-1. License B の「成果物」とは具体的に何ですか？

LICENSE.md 第0章0.6 の定義により、成果物とは「本ソフトウェアを使用することによって生成、出力、または導出した一切の情報、データ、コード、モデル、文書、その他の電子的生成物」です。

本ソフトウェアがどのような機能を持つかによって成果物の性質は異なります。例：

- コードジェネレーターなら → 生成されたコード
- AIトレーニングツールなら → 訓練されたモデルの重み
- データ処理ツールなら → 処理結果のデータ
- ドキュメント生成ツールなら → 生成された文書

ただし、**利用者固有の入力データそのもの**は成果物に含まれません（0.6(a)）。

### Q4-2. License B と License A（AGPL）の公開義務の違いは？

重要な違いは**公開対象**です。

| | License A (AGPL) | License B |
|---|---|---|
| 公開対象 | 本ソフトウェア自体の改変部分 | 本ソフトウェアを使って生成した成果物 |
| トリガー | 改変版の配布／ネットワーク提供 | 商用利用の開始 |
| 公開形式 | AGPL-3.0 の下 | 指定オープンソースライセンスの下 |

License A は「本体のソースコード」の話、License B は「本体を使って作った成果物」の話で、**対象が全く異なります**。

### Q4-3. 私の会社の秘密情報や顧客データが成果物に入っています。公開できません。どうすれば？

以下のいずれかの対応となります。

1. **部分的公開**：LICENSE.md 2.4.6 により、公開不能な情報を含む部分は公開義務から除外できます。ただし、これは「その特定部分が第三者権利等により公開不能」な場合であり、単に「秘密にしたい」理由では認められません。

2. **成果物の主要部分が公開不能な場合**：LICENSE.md 2.4.6 但し書きにより、**License C（サブスク）への移行が必要**です。

3. **入力データとの分離**：成果物から、公開対象（本ソフトウェアの処理結果として導出されたもの）と、非対象（入力データそのもの）を明確に分離してから公開する。

実務的には、顧客データを扱う商業プロジェクトは **License C を選ぶのが通常**です。

### Q4-4. 成果物をMITライセンスで公開しました。これでLicense Bの義務を果たしたことになりますか？

はい。MIT License は LICENSE.md 2.4.1 の指定ライセンスの一つなので要件を満たします。他には Apache-2.0、GPL-3.0、AGPL-3.0、LGPL-3.0、MPL-2.0 が指定されています。

ただし、公開媒体（2.4.2）、タイミング（30日以内、2.4.3）、期間（12ヶ月継続、2.4.4）、通知義務（30日以内、2.4.5）のすべてを満たす必要があります。

### Q4-5. 成果物公開のための通知はどうやってすればいいですか？

LICENSE.md 2.4.5 に従い、公開開始から 30 日以内に、以下の情報を電子メールでライセンサーに通知します。

- 公開URL（GitHub リポジトリ等）
- 適用したライセンス（MIT、Apache-2.0 等）
- 公開開始日
- 利用者（あなた／貴社）の特定情報

通知先：legal@aquaxis.com

### Q4-6. 1年経過したら公開を取り下げてもいいですか？

はい、LICENSE.md 2.4.4 により、公開期間は最低12ヶ月です。12ヶ月経過後は公開を停止できます。ただし、以下の点に留意してください。

- 12ヶ月の起算は「公開開始日」から
- 本ソフトウェアの使用を12ヶ月以内に停止しても、12ヶ月の公開義務は継続
- 公開取り下げは、License B ライセンスの終了を意味するものではない

### Q4-7. 本ソフトウェアで生成したコードを、顧客のクローズドソース製品に組み込んで納品したいです。License Bで可能ですか？

**できません**。License B は成果物の公開義務を課すため、その成果物を顧客がクローズドに保つことはできません。

このケースでは **License C（サブスク）** が必要です。License C は成果物の非公開権（3.3条）を付与するため、顧客のクローズドソース製品に成果物を組み込むことが可能です。

## 5. 技術シナリオ編：具体的な使い方と義務

### Q5-1. 本ソフトウェアをDockerコンテナに入れて配布します。AGPLの影響は？

Docker コンテナ化自体は「改変」ではなく「パッケージング」なので、AGPLの改変トリガーには該当しません。ただし、

- コンテナを第三者に配布する場合は AGPL の表示義務（著作権表示、ライセンス通知）を維持する
- コンテナ内で本ソフトウェアを改変している場合、その改変部分は AGPL の配布／ネットワーク提供トリガーの対象
- コンテナをサーバー上で動かしてネットワーク経由でユーザーに提供する場合、AGPL 第13条が適用される

つまり、**コンテナ化自体は問題ないが、中身の改変とネットワーク提供の有無を見る**のが正しい判断軸です。

### Q5-2. 本ソフトウェアのREST APIを自社アプリから呼び出すだけです。影響は？

未改変の本ソフトウェアをAPI経由で呼び出すだけなら、**あなたのアプリコードは AGPL の影響を受けません**。

AGPLのコピーレフトは「本ソフトウェアを改変した場合」に発動するもので、「本ソフトウェアを呼び出すだけ」のアプリに波及しません。一般にAPI呼び出しは派生物とみなされないため、あなたのアプリは独立した著作物として扱われます。

ただし、**あなたの利用が商用である場合、そもそも License A を選択できません**。その場合の質問は Q2-6 などを参照してください。

### Q5-3. 本ソフトウェアをライブラリとして静的リンクしました。派生物になりますか？

一般論として、静的リンクは「結合著作物」とみなされやすく、AGPL のコピーレフト対象となる可能性が高いです。結果、あなたのアプリケーション全体が AGPL の下で配布される義務が生じる可能性があります。

このリスクを避けたい場合、**動的リンク**（Shared Library / DLL）や**プロセス間通信**（API、IPC、gRPC）を使うか、**License B または License C** への移行を検討してください。

本ソフトウェアをライブラリ形態で商用利用する場合は、**License C のサブスクを推奨**します。サブスクライセンシーには成果物の非公開権があるため、組み込み利用が安全です。

### Q5-4. プラグイン（アドオン）を開発します。私のプラグインはAGPL化されますか？

プラグインが AGPL の影響を受けるかは、技術構成と法的解釈に依存します。

- **独立プロセスで動作、標準APIで通信**：独立著作物とみなされる可能性が高い → AGPL化しない可能性
- **本ソフトウェアの内部APIを直接呼ぶ、内部データ構造に依存**：派生物とみなされる可能性が高い → AGPL化する可能性

Linux 等の多くのAGPL/GPLプロジェクトは、プラグインを独立著作物として扱う方針を表明することが多いですが、**本プロジェクトの方針は AQUAXIS TECHNOLOGY に個別確認**してください。

### Q5-5. 本ソフトウェアと通信する別ソフトを作ります。別ソフトもAGPLですか？

プロセス境界を越えてネットワークまたはIPCで通信する場合、一般に別ソフトは独立著作物とみなされ、AGPLの影響を受けません。

ただし、FSFは「密結合な通信」や「データ構造の共有」がある場合、派生物とみなす立場を取ることがあります。判断に迷う場合は弁護士にご相談いただくか、License C をご検討ください。

### Q5-6. 本ソフトウェアを fork して、まったく別のプロジェクトとして公開します。可能ですか？

はい、**AGPLの範囲内で可能**です。Fork して公開する場合、

- forkした側のプロジェクトは AGPL-3.0 のまま公開する義務
- License B または License C として再配布する権限はない（CLAがないため）
- ライセンサーの商標（製品名、ロゴ）は使用不可（LICENSE.md 2.8条）

商標上の混同を避けるため、**forkプロジェクトは独自の名称**を付けてください。

### Q5-7. AIの学習データとして本ソフトウェアのソースコードを使ってもいいですか？

これはAGPLが想定していなかった新しい論点で、業界全体で議論中です。一般論として、

- ソースコードを**学習データとして取り込み、モデルに内在化させる行為**は、複製・翻案に該当する可能性があり、AGPLの適用対象となる可能性
- 学習済みモデルが AGPL コードを「記憶」して出力する場合、AGPL義務が派生する可能性

本プロジェクトは、AI学習利用について**事前に書面での許可を求める**立場を取ります。商用AI学習利用については License C（個別条項）が必要になります。

## 6. ライセンス移行編：状況が変わったら

### Q6-1. 非商用で使っていましたが、起業して商用化します。どう移行すればいいですか？

商用利用開始時点で、License B または License C を選択する必要があります（LICENSE.md 1.6条）。

重要なポイント：

- **商用化前に非商用として行った改変・配布は引き続きAGPL-3.0**（遡及しない）
- **商用化開始日以降**の利用から、License B の成果物公開義務、または License C のサブスク料金が発生
- 商用化の定義は LICENSE.md 0.1 を参照（売上発生 or 売上見込み）

## 8. コントリビュータ編：CLA について

### Q8-1. なぜ CLA に署名しないと PR をマージできないのですか？

本プロジェクトはデュアルライセンス構造のため、ライセンサーはコントリビュータから「License A に加えて License B/C としても再ライセンスできる権利」を取得する必要があります。これを法的に保証するのが CLA です。CLAなしでは、あなたの貢献部分だけ License B/C で配布できないという事態が発生します。

### Q8-2. CLA に署名すると、著作権を放棄することになりますか？

**いいえ、著作権は放棄しません**。CLA（I-4条）により、あなたの著作権は保持されます。ライセンサーに対して広範な利用・再ライセンス権を付与するだけです。

### Q8-3. CLA 署名後、心変わりして貢献を撤回できますか？

将来の貢献については、CLA の終了を書面で通知することで撤回できます。ただし、**既に提出した貢献のライセンス付与は永続的・取消不能**（I-8条）です。これは、プロジェクトがコントリビュータの事後撤回により法的に不安定になることを防ぐためです。

### Q8-4. 会社員ですが、業務時間外に個人として貢献したいです。CLA はどうすれば？

CLA I-7 のチェックボックスで、雇用主の権利が一切含まれないことを確認する選択肢を選びます。ただし、

- 会社のPC・ネットワークを使っていないこと
- 業務時間外に行っていること
- 会社の機密情報・業務知識を使っていないこと
- 雇用契約上、個人的な創作物が会社に帰属しないこと

を自己確認してください。不明な場合は、事前に会社の法務・コンプライアンス部門に確認することを強く推奨します。

### Q8-5. 会社から CLA（CCLA）に署名してもらうには、どう進めればいいですか？

以下のプロセスが一般的です。

1. 会社の法務・コンプライアンス部門に本CCLAドラフトを提示
2. 会社内で指定貢献者のリストを作成（社員の氏名、メール、GitHub ユーザー名）
3. 会社の権限者（役員・管理者）が CCLA に記名押印または電子署名
4. 署名済み文書と指定貢献者リストをライセンサーに提出

### Q8-6. CCLA 署名後、新しい社員を追加したいのですが？

CCLA II-6 に従い、連絡担当者から電子メールでライセンサーに通知することで追加できます。新たな CCLA 署名は不要です。

### Q8-7. バグ報告や機能提案も CLA の対象ですか？

CLA の「貢献」（共通定義(b)）には、Issue報告、機能提案、ドキュメント改善、翻訳なども含まれます。ただし、ソースコードを伴わない単純なバグ報告・機能提案は、実務上 CLA 署名なしで受け付けるプロジェクトも多いです。本プロジェクトの運用方針は http://aquaxis.com/community/guidelines を参照してください。

### Q9-1. 会社の法務部から「AGPLは禁止」と言われました。どうすれば？

多くの企業が AGPL の使用を社内ポリシーで制限しています。この場合、以下のいずれかで対応できます。

3. 本ソフトウェアのファイルヘッダーに License C の SPDX 識別子（`LicenseRef-ProductName-Commercial` 等）が記載されていることを確認

これにより、監査レポート上 AGPL 義務の対象外として扱われます。
